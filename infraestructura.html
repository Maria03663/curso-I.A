Máquinas Virtuales y Cloud
Infraestructura GCP, AWS, DigitalOcean, Cloudflare, costos, creación de VM

Una máquina virtual (VM) es un entorno de computación virtualizado que emula un servidor físico completo, permitiéndote ejecutar tu propio sistema operativo y aplicaciones en la nube. Es el bloque fundamental para desplegar modelos de ML en producción.

Cada proveedor cloud tiene sus ventajas: GCP ofrece $300 en créditos gratuitos y excelente integración con Vertex AI; AWS es el más maduro con EC2 y SageMaker; DigitalOcean es simple y económico para proyectos chicos; Cloudflare destaca en edge computing con Workers. La elección depende de tu presupuesto, escala y familiaridad.

Costos estimados: Una VM básica (2 vCPU, 4 GB RAM) cuesta ~$25-40/mes en GCP/AWS, ~$12-24/mes en DigitalOcean. Siempre configurá alertas de presupuesto para evitar sorpresas. El free tier de GCP incluye una instancia e2-micro gratuita de por vida.

Crearemos una VM en GCP paso a paso: desde la consola, seleccionamos Compute Engine → Crear instancia, elegimos región (us-east1 por latencia), tipo e2-medium (2 vCPU, 4 GB), Ubuntu 22.04 LTS, permitimos tráfico HTTP/HTTPS, y generamos una clave SSH para conectarnos.

FastAPI desde Cero
Backend Instalación, endpoints, Pydantic, Swagger, predicción ML

FastAPI es uno de los frameworks web más rápidos de Python, ideal para servir modelos de ML gracias a su alto rendimiento asíncrono, validación automática con Pydantic y documentación interactiva con Swagger UI. Se instala con pip install fastapi uvicorn.

Creamos endpoints básicos: un GET /health que devuelve {"status": "ok"}, y un POST /predict que recibe features de entrada y retorna la predicción del modelo. Usamos modelos Pydantic para definir esquemas de request/response con tipos estrictos — si los datos no coinciden, FastAPI devuelve error 422 automáticamente.

Ejemplo de código:
from fastapi import FastAPI
from pydantic import BaseModel
import joblib

app = FastAPI()
model = joblib.load("modelo_entrenado.pkl")

class Features(BaseModel):
    sepal_length: float
    sepal_width: float
    petal_length: float
    petal_width: float

@app.post("/predict")
def predict(features: Features):
    X = [[features.sepal_length, features.sepal_width, features.petal_length, features.petal_width]]
    pred = model.predict(X)[0]
    return {"prediction": int(pred)}

Al correr uvicorn main:app --reload, visitás http://localhost:8000/docs y ves Swagger UI con todos tus endpoints documentados automáticamente, listos para probar.

Docker y Contenedores
Contenedores Dockerfile para FastAPI, build, run, docker-compose

Contenedores vs VMs: una VM virtualiza hardware completo e incluye su propio sistema operativo huésped, lo que la hace pesada (varios GB). Un contenedor comparte el kernel del host y solo empaqueta la aplicación y sus dependencias (cientos de MB). Los contenedores arrancan en segundos, las VMs en minutos.

El Dockerfile define cómo construir la imagen de tu app de FastAPI. Partimos de python:3.11-slim, copiamos requirements.txt, instalamos dependencias, copiamos el código fuente, y definimos el comando de inicio con CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]. Cada instrucción genera una capa cacheable para builds más rápidos.

Ejemplo práctico: docker build -t ml-api:v1 . construye la imagen. docker run -d -p 8000:8000 --name ml-api ml-api:v1 la ejecuta en segundo plano mapeando el puerto. docker-compose permite definir múltiples servicios (API + Redis + PostgreSQL) en un solo archivo YAML y levantarlos con docker-compose up -d.

Buenas prácticas: usá .dockerignore para excluir __pycache__, .git y archivos grandes; mantené las imágenes livianas con slim o alpine; nunca almacenes secretos en la imagen; usá variables de entorno para configuración.

CI/CD y Monitoreo
DevOps GitHub Actions, deploy automático, health checks, logs, rollback

GitHub Actions permite automatizar el pipeline de CI/CD directamente desde tu repositorio. Definimos un workflow en .github/workflows/deploy.yml que se dispara con cada push a main: corre tests, construye la imagen Docker, la pushea a un container registry (Docker Hub o GitHub Container Registry), y hace SSH a la VM para actualizar el contenedor.

El deploy automático funciona así: GitHub Actions se conecta por SSH a tu VM en GCP usando secretos (SSH_PRIVATE_KEY, HOST, USERNAME), pullea la nueva imagen con docker pull, detiene el contenedor viejo, y levanta el nuevo con docker run. El downtime es de solo 2-3 segundos si usás la estrategia correcta.

Estrategias de rollback: Siempre tagueá tus imágenes con el SHA del commit (ml-api:abc123). Si el deploy falla, volvés a la imagen anterior con docker run ml-api:def456. Implementá health checks (GET /health) que el balanceador consulta cada 10s; si fallan 3 veces seguidas, marcá la instancia como unhealthy y redirigí tráfico.

Configurá logs centralizados con docker logs + journald o Cloud Logging de GCP. Para monitoreo, Prometheus + Grafana te dan dashboards en tiempo real de latencia, tasa de errores y uso de recursos.
